# Sagittarius LLM + YOLO 方块抓取操作指南

## 1. 项目目标

在已经打通的“大模型文本指令 -> JSON 规划 -> MoveIt 执行”链路上，增加一个基于预训练 YOLO 模型的视觉模块，实现下面这条完整流程：

1. 相机采集图像
2. YOLO 检测方块
3. 将检测框中心像素坐标换算为机械臂基座坐标
4. 机械臂移动到目标上方并抓取
5. 抬升后移动到 `place_ready`
6. 打开夹爪完成放置

这条链路已经以 `sagittarius_llm_control` 包为核心落地，当前版本的重点是先把“单目标方块抓取与放置”跑稳，再逐步扩展到多目标、重试、状态记忆等高级能力。

## 2. 项目规划

建议按下面四个阶段推进项目：

### 第一阶段：打通视觉抓取闭环

目标：

- YOLO 能稳定输出方块检测框
- 能通过标定把像素坐标转成机械臂基座坐标
- 机械臂能完成一次“抓取 -> 搬运 -> 放置”

交付物：

- `/sgr532/detect_block` 视觉服务
- `pick_detected_block` 执行步骤
- 完整的 `llm_task_system.launch`

### 第二阶段：让大模型能调用视觉抓取能力

目标：

- 用户说“抓取方块并放到 place_ready”
- LLM 规划器输出 `pick_detected_block`
- 本地执行器调用视觉服务并完成抓取

交付物：

- 更新后的 planner prompt
- `planner.py` 中对 `pick_detected_block` 的规范化
- `executor.py` 中的视觉抓取执行逻辑

### 第三阶段：工程化和易用性

目标：

- 一条 launch 命令启动主要链路
- 清晰的配置项和默认参数
- 清晰的故障提示和调试顺序

交付物：

- `runtime.yaml`
- `操作指南.md`
- 更一致的服务命名和依赖声明

### 第四阶段：后续扩展方向

建议扩展顺序：

1. 增加抓取失败后的自动重试
2. 支持多个目标类别和优先级策略
3. 支持多目标连续抓取
4. 引入任务状态记忆
5. 加入仿真回归测试

## 3. 当前代码结构

当前实现集中在：

`src/sagittarius_ws/src/sagittarius_arm_ros/sagittarius_llm_control`

关键文件分工如下：

- `scripts/llm_task_server.py`
  暴露自然语言任务服务 `/sgr532/execute_task`
- `scripts/llm_task_cli.py`
  命令行客户端，用于发送自然语言任务
- `scripts/yolo_block_detector.py`
  YOLO 检测节点入口
- `src/sagittarius_llm_control/planner.py`
  调用 DeepSeek，把自然语言转为结构化 JSON 步骤
- `src/sagittarius_llm_control/executor.py`
  执行 JSON 步骤，负责 MoveIt、夹爪、视觉抓取流程
- `src/sagittarius_llm_control/vision_detector.py`
  视觉服务核心，完成图像订阅、YOLO 推理、像素到基座坐标换算
- `src/sagittarius_llm_control/vision_client.py`
  执行器侧的视觉服务客户端
- `config/runtime.yaml`
  运行时主配置，包括安全边界、视觉参数、抓取高度和命名点
- `launch/llm_task_system.launch`
  启动 MoveIt、相机、YOLO 节点、LLM 服务的总入口
- `srv/DetectBlock.srv`
  视觉检测服务接口
- `srv/ExecuteNaturalLanguageTask.srv`
  自然语言任务服务接口

## 4. 已完成的核心能力

当前代码已经支持：

- 基于 DeepSeek 的自然语言任务规划
- JSON DSL 规范化和安全检查
- MoveIt 位姿执行和命名位姿执行

本轮补充完成的工程化修复包括：

- `package.xml` 补充了 `python3-opencv` 和 `python3-yaml` 运行依赖

## 5. 整体执行流程

系统的逻辑链路如下：

```text
用户自然语言
  -> llm_task_cli.py
  -> /sgr532/execute_task
  -> planner.py 调用 DeepSeek
  -> 生成 JSON 计划
  -> executor.py 执行
  -> 如果步骤为 pick_detected_block
  -> 调用 /sgr532/detect_block
  -> vision_detector.py 用 YOLO 检测方块
  -> 像素坐标转基座坐标
  -> MoveIt 到 pregrasp_z / pick_z / lift_z
  -> 移动到 place_ready
  -> 打开夹爪放置
```

## 6. 视觉抓取动作的本地执行顺序

`pick_detected_block` 当前在执行器里的顺序是：

1. 移动到 `vision.observation_waypoint`
2. 调用视觉服务 `/sgr532/detect_block`
3. 选择目标检测框
4. 将中心像素坐标换算为基座坐标
5. 打开夹爪
6. 移动到目标上方 `pregrasp_z`
7. 下探到 `pick_z`
8. 闭合夹爪
9. 抬升到 `lift_z`
10. 移动到 `place_ready`
11. 打开夹爪释放方块

## 8. 标定前提

当前坐标换算使用的是线性回归标定参数：

`src/sagittarius_ws/src/sagittarius_arm_ros/sagittarius_perception/sagittarius_object_color_detector/config/vision_config.yaml`

视觉服务会读取其中的：

- `LinearRegression.k1`
- `LinearRegression.b1`
- `LinearRegression.k2`
- `LinearRegression.b2`

换算方式为：

```text
base_x = k1 * center_v + b1
base_y = k2 * center_u + b2
base_z = vision.planar_z
```

注意：

- 这是平面标定，不是完整 3D 视觉定位
- 如果相机高度、角度、安装位置变化了，必须重新标定
- `planar_z` 需要与你们桌面和目标高度匹配
- 如果抓取点存在稳定的系统偏差，可以先通过 `vision.position_offset_xyz` 做小范围补偿

## 9. 关键配置文件

vision:
  enabled: true
  service_name: /sgr532/detect_block
  image_topic: /usb_cam/image_raw
  model_path: ""
  calibration_file: /home/robotics/team8/src/sagittarius_ws/src/sagittarius_arm_ros/sagittarius_perception/sagittarius_object_color_detector/config/vision_config.yaml

  observation_waypoint: observation
  detect_from_current_pose_first: false
  place_target_name: place_ready
  grasp_orientation_preset: vertical_down

  pick_z: 0.035
  lift_z: 0.18
  gripper_open_command: open
  gripper_close_command: close

    cube: block
    cuboid: block

    carton: block

最需要根据现场修改的是：

- `vision.model_path`
- `vision.calibration_file`
- `vision.pregrasp_z`
- `vision.pick_z`
- `vision.lift_z`
- `vision.position_offset_xyz`
- `vision.image_topic`
- `vision.detect_from_current_pose_first`
- `vision.default_target_label`
- `vision.label_aliases`

### 9.3 命名位姿

```yaml
named_waypoints:
  observation:
    position: [0.18, 0.00, 0.22]
    orientation_preset: vertical_down
  pick_ready:
    position: [0.1253, 0.0780, 0.1554]
    orientation_preset: vertical_down
  pick_ready_2:
    position: [0.1550, 0.1300, 0.1554]
    orientation_preset: vertical_down
  pick_ready_3:
    position: [0.1550, -0.0600, 0.1554]
    orientation_preset: vertical_down
  place_ready:
    position: [0.18, -0.14, 0.12]
    orientation_preset: vertical_down
  place_ready_2:
    position: [0.2200, 0.0000, 0.1200]
    orientation_preset: vertical_down
  place_ready_3:
    position: [0.2200, 0.1400, 0.1200]
    orientation_preset: vertical_down
```

建议：

- `observation` 设成适合相机观测整个工作区的位置
- 如果相机装在机械臂末端并需要俯视桌面，`observation.orientation_preset` 应优先使用 `vertical_down`
- 如果机械臂默认起始姿态看不到桌面，应将 `vision.detect_from_current_pose_first` 设为 `false`，让系统先移动到 `observation` 再检测
- `place_ready` 设成安全且固定的放置点
- `place_ready_2`、`place_ready_3` 可以作为额外的放置点，例如中间料框、左侧料框、右侧料框
- `pick_ready` 可以保留作为手工调试位姿
- `pick_ready_2`、`pick_ready_3` 可以作为额外的手工调试或预抓取位姿

## 17. 常见问题

### 17.1 `ultralytics is not installed`

说明 YOLO 依赖未安装。

解决：

```bash
python3 -m pip install ultralytics torch torchvision opencv-python
```

### 17.2 `vision.model_path is empty`

说明没有把 `.pt` 模型路径传给视觉节点。

解决：

- 在 launch 参数中传 `vision_model_path:=...`
- 或者直接在 `runtime.yaml` 中写入真实路径

### 17.3 `Calibration file does not exist`

说明标定文件路径不对。

请检查：

- `vision.calibration_file`
- launch 中的 `vision_calibration_file`

### 17.4 `No image has been received from the camera yet`

说明相机没有正常出图，或者 `vision.image_topic` 配错。

先检查：

```bash
rostopic list | grep image
rostopic echo -n 1 /usb_cam/image_raw/header
```

### 17.5 `No detection matched target label`

说明 YOLO 没有识别到指定类别，或者类别名与配置不匹配。

请检查：

- 模型是否真的能检测到目标
- `vision.detection_labels`
- `vision.label_aliases`
- `vision.default_target_label`

### 17.6 `timeout exceeded while waiting for service /sgr532/execute_task`

说明 LLM 服务还没起来，或者 ROS master 不在线。

先检查：

```bash
source ~/team8/devel/setup.bash
rosservice list | grep execute_task
```

如果提示 `Unable to communicate with master`：

- 启动 `roscore`
- 或重新执行 `roslaunch sagittarius_llm_control llm_task_system.launch`

### 17.7 `Translation delta ... exceeds limit ...`

旧版本执行器会在单步位移超限时直接报错。

当前版本已经改成自动分段执行。如果你还看到这个报错，通常是：

- 还没有重新 `catkin_make`
- 还没有重新 `source devel/setup.bash`
- 实际运行的仍是旧的安装脚本

### 17.8 抓取点明显偏移

大概率是标定问题，而不是 YOLO 本身问题。

请检查：

- 相机是否移动过
- 相机是否倾斜角变化
- 标定文件是否为当前安装姿态生成
- `planar_z` 是否合理

如果偏移方向稳定一致，可以先用 `vision.position_offset_xyz` 进行补偿：

- 抓取点偏后或偏前：优先调整 `position_offset_xyz[0]`
- 抓取点偏左或偏右：优先调整 `position_offset_xyz[1]`
- 抓取高度偏高或偏低：优先调整 `position_offset_xyz[2]`

建议每次只改一个轴，每次改 `0.005` 到 `0.015` 米，重新启动后再试。

## 18. 安全建议

建议先以低速运行：

- `runtime.velocity_scaling: 0.2`
- `runtime.acceleration_scaling: 0.2`

并遵守下面原则：

- 先测 `--plan-only`
- 先单独测视觉服务
- 先空夹爪测试
- 初次测试时把抓取高度和桌面留出安全余量
- 确认 `place_ready` 周围没有障碍物
